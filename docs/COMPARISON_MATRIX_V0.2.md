# Aster Runner v0.2 comparison matrix

| System | Runtime / isolation | V8 question | Capability model | Database / state model | Recovery story | Where Aster differs |
|---|---|---|---|---|---|---|
| Convex in-process OSS runner | V8 isolates inside the deployment backend process; limited by configured V8 worker threads | Uses async Promise syscalls for db operations inside one runner process | Backend process has database transaction authority and execution authority | Transaction object remains in backend; single writer lease protects commits | Backend retries queries/mutations; process restart loses in-flight work | Aster moves execution out to cells and removes DB credentials from cells while keeping Convex as writer |
| Convex Funrun-style remote runner | Remote Rust/V8 execution service; proprietary server reference | Public descriptions imply normal V8 async execution and remote snapshot loading | Runner tier can load/read snapshots; backend commits | Read-only execution replicas return outcomes/read-write info to backend | Overload retries, remote runner failover, backend committer remains truth | Aster adds capsule broker and read traps so cells are not read replicas |
| Cloudflare workerd / Workers | V8 isolates in a shared runtime with strong production hardening | Does not suspend arbitrary stacks for host hydration; Workers use event loop and async host APIs | Service bindings, Durable Object bindings, environment capabilities | External state through KV/D1/R2/Durable Objects; not an MVCC app DB transaction model | Isolate eviction/restart; platform retries according to event semantics | Aster borrows isolate density but makes MVCC snapshot capsules the explicit data capability |
| Cloudflare Durable Objects | Single-threaded object instances with colocated durable state | V8 async event model; object methods await storage/network | Object ID routes authority to one object; bindings mediate access | Per-object serialized state, not general cross-table MVCC | Object can be evicted/restarted; storage is durable | Aster keeps Convex's single writer per deployment but scales stateless function execution cells |
| Fermyon Spin | WebAssembly components, WASI, fast cold starts | No V8; continuation would be WASM async hostcall/future problem | WASI capabilities and component interfaces | External DB/key-value through host components | Instance restart; host retries per trigger | Aster targets existing Convex JS/V8 semantics; WASM is a possible future cell backend |
| Shopify Functions | WebAssembly sandbox for deterministic commerce extensions | No V8; strict deterministic function interface | Host passes bounded input; function returns bounded output; no arbitrary DB access | Snapshot-like input payload, no interactive MVCC reads | Rerun function on request; no side effects inside function | Aster's capsules resemble Shopify's bounded input, but Aster adds interactive read traps and OCC writes |
| Fly Machines / Firecracker-style microVMs | Process/VM isolation, fast-ish machine start, can run anything | V8 is inside guest if used; no special continuation support | VM boundary; secrets/network per machine | App-defined; broad authority if VM has DB creds | Restart machine, reschedule, persistent volumes optional | Aster chooses long-lived tenant cells rather than per-invocation VMs to keep Convex latency/density |
| Deno Deploy / deno_core apps | V8 via deno_core ops and async resources | Async ops are first-class; Promise resolvers and microtasks drive host continuations | Permissions/ops mediate FS/network/env | External services; not a built-in MVCC committer | Isolate restart; platform-level retry | Aster's V8 proof is closer to deno_core ops than to stack capture, but with MVCC capsule seals |
| Antithesis / deterministic replay systems | Deterministic execution and replay harnesses | V8 may be part of replayed process, but goal is debugging/failure reproduction | Not primarily a capability system | Captures execution for replay, not live DB authority separation | Replay to reproduce failures | Aster could use replay for debugging cell crashes, but correctness is online OCC + seals |
| FoundationDB | Deterministic transaction system with optimistic concurrency | No V8 runtime claim | Clients have DB credentials; tenant isolation via database/security layers | Serializable transactions over key-value store | Transactions retry on conflicts; replicated log recovers | Aster borrows OCC discipline but narrows runner authority with capsules |
| TigerBeetle | Deterministic replicated state machine for financial transactions | No V8 | Clients submit commands; replicas validate deterministic state transitions | Single-purpose ledger database | Consensus/replica recovery; no arbitrary side effects | Aster's effect ledger idea borrows the “make ambiguity impossible” posture for actions |

## Notes

- The v0.2 V8 result moves Aster closer to Deno/Convex internals than to Scheme-style captured continuations. The continuation is a Promise job, not a serialized native stack.
- Systems such as Shopify Functions prove the value of bounded input capabilities, but they avoid interactive reads. Aster's novelty remains the fallback trap protocol over MVCC snapshots.
- Workerd is the best prior art for dense V8 multi-tenancy; it does not solve Convex's transaction/read-set problem by itself.
